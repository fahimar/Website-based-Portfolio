Docker Series · Part 01

Docker + Postgres:
Volume কি সত্যিই Temporary?

initdb কেন বারবার run করে না — এবং production এ কী করা উচিত

ফাহিম আল রশিদ · March 2026 · 8 min read · Docker · Postgres · DevOps

Docker volume temporary না — persistent। Container delete হলেও data থাকে। কিন্তু /docker-entrypoint-initdb.d/ শুধু প্রথমবার empty volume দেখলে run করে। Production এ schema update করতে migration tool ব্যবহার করো — initdb এর উপর ভরসা করো না।

প্রশ্নটা কোথা থেকে এলো?

একটা common scenario। তুমি Postgres container চালাচ্ছো, নতুন একটা table যোগ করলে schema.sql এ, container restart দিলে — কিন্তু table নেই। মাথায় প্রশ্ন আসে: volume কি delete হয়ে গেছে? নাকি schema run হয়নি?

এই confusion টা আসলে Docker এর একটা intentional design decision না বোঝার কারণে হয়। চলো step by step ভেঙে দেখি।

Volume কি আসলে Temporary?

না। Docker volume হলো persistent storage। এটা container এর lifecycle এর সাথে tied না। মানে:

✓
VOLUME TRUTH

Container stop করো, restart করো, এমনকি docker compose down করো — volume এর data থাকবে। Volume আলাদা entity, container না।

তাহলে সমস্যা কোথায়?

initdb.d — The One-Time Magic

Postgres Docker image এর একটা feature আছে: /docker-entrypoint-initdb.d/ ফোল্ডারে যেকোনো .sql বা .sh ফাইল রাখলে সেটা startup এ run করে।

কিন্তু শুধুমাত্র তখনই যখন data directory সম্পূর্ণ empty থাকে।

Trigger
docker compose up (প্রথমবার)
↓
Check
postgres_data volume → empty?
↓
Action (empty হলে)
initdb.d scripts run → tables create → DB initialize
↓
Result (পরের বার)
volume already initialized → initdb.d SKIP ❌

এটা intentional। Postgres ধরে নেয়: "data already আছে, আমি overwrite করবো না।" এটা data safety feature।

তোমার Real-World Scenario

ধরো তোমার project এ এই tables ছিল:

-- schema.sql (original)
CREATE TABLE faq_entries (...);
CREATE TABLE training_entries (...);
CREATE TABLE conversation_logs (...);

পরে তুমি যোগ করলে:

CREATE TABLE groq_usage_logs (...); -- নতুন

কিন্তু docker compose up করার পর groq_usage_logs নেই।

⚠️
WHY IT HAPPENED

Volume আগে থেকেই initialized ছিল (তিনটা table নিয়ে)। Postgres দেখলো "DB exists" — তাই initdb.d skip করলো। নতুন table কোনোদিনই create হয়নি।

Volume এর সত্যিকারের Life Cycle

Action Volume Data Status
docker compose stop সব data থাকে ✓ SAFE
docker compose down সব data থাকে ✓ SAFE
Container delete সব data থাকে ✓ SAFE
docker compose down -v Volume সহ সব যায় ✗ DANGER
docker volume rm <name> Volume delete, সব যায় ✗ DANGER
🔥
PRODUCTION WARNING

docker compose down -v flag টা production এ কখনো blindly run করো না। এই -v flag টাই সব data মুছে দেয়।

Volume আছে কিনা Check করো

# সব volume দেখো
docker volume ls

# Project এর volume inspect করো
docker volume inspect faces-chatbot_postgres_data

Volume এর নাম সাধারণত হয়: <project-folder-name>_<volume-name-in-compose>।

যদি docker-compose.yml এ লেখা থাকে:

volumes:
  postgres_data:

আর project folder এর নাম faces-chatbot হলে, volume এর নাম হবে faces-chatbot_postgres_data।

Production-Safe Solution: Migration

initdb.d দিয়ে শুধু initial setup করা যায়। Schema evolve করতে হলে দরকার migration system।

Concept টা simple:

  • 1
    প্রতিটা schema change কে numbered file এ রাখো (e.g., V1__initial.sql, V2__add_groq_logs.sql)
  • 2
    Migration tool track করে কোন version পর্যন্ত applied হয়েছে
  • 3
    শুধু নতুন files apply করে — আগেরগুলো skip করে

// Popular Migration Tools

Alembic

SQLAlchemy projects এর জন্য। Python-based। Auto-generate migrations সহ।

Flyway

SQL-first approach। Java ecosystem popular। Docker image আছে।

Liquibase

XML/YAML/JSON format এ schema define করো। Enterprise-friendly।

Manual Versioned SQL

Simplest। migrations/ folder এ numbered SQL files। Small projects এর জন্য perfect।

// Quick Fix (Immediate Solution)

এখনই production এ table add করতে চাইলে, সরাসরি running container এ run করো:

# Running container এ connect করো
docker exec -it <postgres-container-name> psql -U <user> -d <dbname>

# তারপর SQL run করো
CREATE TABLE groq_usage_logs (
  id SERIAL PRIMARY KEY,
  created_at TIMESTAMP DEFAULT NOW()
  -- বাকি columns...
);
💡
PRO TIP

এই manual change টাকেও একটা migration file এ document করে রাখো। নাহলে পরে নতুন environment setup করতে গেলে এই table miss হবে।

Docker Compose env_file Bonus Fix

যেহেতু কথা উঠলো — একটা common compose error এর সমাধানও দিয়ে রাখি।

Error: services.api.environment must be a mapping

এটা হয় যখন environment: এ `.env` content paste করা হয় যেটা YAML mapping format এ নেই।

# ❌ Wrong — string format
environment:
  - DATABASE_URL=postgres://...

# ✅ Correct — use env_file instead
api:
  build: .
  env_file:
    - .env.prod
  depends_on:
    - postgres

env_file: ব্যবহার করলে environment: block দরকার নেই।
